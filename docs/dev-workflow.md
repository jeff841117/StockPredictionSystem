# 開發協作流程

## 1. 文件目的

這份文件用來定義「台股分析與模擬交易系統」在 Codex 中的協作方式。  
目標是讓專案可以用固定流程持續開發、逐步學習，並保留清楚的 GitHub 紀錄。

## 2. 協作核心原則

1. 以 Python 為主要開發技術
2. 先規格、再開發、後驗收
3. 每次只處理一個小功能
4. 每個功能都要能獨立驗收
5. 每個功能盡量獨立 commit / push
6. AI 以教學輔助、拆解、檢查、修正為主
7. 不直接依賴 AI 一次產出完整成品

## 3. Thread 分工

建議使用 4 個 thread：

1. `控制台`
2. `規劃`
3. `開發`
4. `驗收`

### `控制台`

負責：

1. 接收使用者需求
2. 宣告本輪目標
3. 協調規劃、開發、驗收三個 thread
4. 收斂目前狀態與下一步
5. 管理版本節奏與功能交接

不負責：

1. 長篇規格細節撰寫
2. 細部程式實作
3. 長期保存每一輪技術討論細節

### `規劃`

負責：

1. 整理功能需求
2. 切分功能範圍
3. 定義驗收標準
4. 寫功能規格草案

不負責：

1. 直接寫完整功能程式
2. 自行更改專案主方向

### `開發`

負責：

1. 依規格實作功能
2. 回報已完成項目
3. 回報風險、阻塞與未完成部分
4. 整理本次修改內容

不負責：

1. 擅自改規格
2. 跳過驗收直接宣布完成

### `驗收`

負責：

1. 依規格檢查完成度
2. 列出通過與未通過項目
3. 整理回修清單
4. 補充測試或風險建議

不負責：

1. 直接改需求方向
2. 代替開發 thread 做架構決策

## 4. 每輪開發標準流程

每一個小功能都盡量走完以下流程：

1. 使用者在 `控制台` 提出本輪需求
2. `控制台` 指派 `規劃` thread 產出功能規格
3. `規劃` thread 交付規格摘要與驗收標準
4. `控制台` 或使用者確認規格是否可行
5. `控制台` 將規格摘要交給 `開發` thread
6. `開發` thread 依規格實作
7. `開發` thread 回報完成內容、風險與修改檔案
8. `控制台` 或 `開發` thread 通知 `驗收` thread 開始檢查
9. `驗收` thread 回報通過項目、未通過項目與回修建議
10. 若驗收通過，進行 commit / push
11. 若驗收未通過，回到 `開發` thread 修正

## 5. 使用者在什麼時候介入

使用者建議在這些節點介入確認：

1. 第一輪專案啟動時
2. 每個功能規格完成後
3. 驗收結果出來後
4. commit / push 前
5. 專案方向有明顯變更時

一開始不建議完全自動化，建議先採用半自動流程。

## 6. 控制台應保存哪些內容

控制台適合保留：

1. 專案總目標
2. 第一版 MVP 範圍
3. 每一輪正在做的功能
4. 各 thread 的交接摘要
5. 目前版本狀態

控制台不適合堆積：

1. 完整長篇規格
2. 所有技術細節
3. 每一次完整驗收全文

## 7. 規格文件的主落點

不同類型文件建議主落在：

1. 需求文件 / 功能規格：`規劃`
2. 技術設計 / 架構 / API：`開發`
3. 驗收清單 / bug 清單：`驗收`
4. 專案總方向 / 版本節奏：`控制台`

## 8. 交接原則

跨 thread 交接時，應只傳摘要，不直接貼整段長對話。

建議交接內容包含：

1. 本次目標
2. 功能範圍
3. 不包含項目
4. 驗收標準
5. 已知風險或限制

## 9. subagent 使用原則

`subagent` 適合處理短期、局部、可平行的任務，例如：

1. 掃描相關檔案
2. 比較兩個方案差異
3. 補查測試缺口
4. 整理風險點

不建議把 `subagent` 當成常駐角色。

## 10. 與文件的關係

本流程建議搭配以下文件一起使用：

1. `project-spec.md`
2. `mvp-plan.md`
3. `project-structure.md`
4. `feature-spec-template.md`
5. `commit-workflow.md`
6. `learning-log.md`

## 11. 每輪結束時建議更新

每完成一個功能後，建議同步更新：

1. 功能規格文件
2. 學習紀錄
3. README 或進度說明
4. GitHub commit 紀錄
