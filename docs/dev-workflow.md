# 開發協作流程

本文件描述目前專案採用的協作節奏，重點是讓 `規劃`、`開發`、`驗收` 與 `控制台` 的交接維持一致。

## 協作角色

本專案固定使用 4 個 thread：

1. `控制台`
2. `規劃`
3. `開發`
4. `驗收`

### `控制台`

- 接收使用者需求
- 宣告本輪目標
- 協調規劃、開發、驗收交接
- 收斂各 thread 結論

### `規劃`

- 整理需求
- 切分功能範圍
- 定義驗收標準
- 輸出可交接給開發的規格摘要

### `開發`

- 僅依規格實作
- 回報完成內容、未完成項目、風險與驗證結果
- 輸出固定開發交付摘要

### `驗收`

- 依驗收標準檢查
- 列出通過與未通過項目
- 提供回修建議
- 判斷是否可進行 commit / push

## 每輪標準流程

1. 使用者或控制台定義本輪目標
2. `規劃` thread 產出 `[規劃交付摘要]`
3. `開發` thread 依規格實作
4. `開發` thread 回報 `[開發交付摘要]`
5. `驗收` thread 依規格檢查
6. 若未通過，回到 `開發` thread 回修
7. 若通過，再進行 commit / push

## 開發 thread 固定輸出格式

目前 `開發` thread 固定使用：

```text
[開發交付摘要]
依據規格：
本次完成內容：
修改檔案：
尚未完成項目：
已知風險：
建議驗收方式：
建議交給驗收 thread 的摘要：
```

用途：

- 方便跨 thread 交接
- 讓驗收 thread 快速對照規格與修改範圍
- 降低冗長過程造成的資訊噪音

## 驗證原則

1. 先做最小必要驗證
2. 優先沿用既有測試方式
3. 若改動程式邏輯，應回報：
   - 執行了哪些測試
   - 哪些已驗證
   - 哪些尚未驗證
4. 若失敗，需區分：
   - 環境問題
   - 相依性問題
   - 程式本身問題

目前專案主要自動化驗證指令：

```bash
python -m unittest discover -s tests -v
```

CI 對應：

1. GitHub Actions 主 workflow 目前也會執行同一條指令
2. 目標是先驗證核心 `unittest` 流程
3. 真實 TWSE smoke test 不納入主 CI，避免外部資料波動影響穩定性

## 文件同步原則

每完成一輪功能，應視需要同步更新：

1. `README.md`
2. `docs/project-structure.md`
3. `docs/mvp-plan.md`
4. `docs/features/` 下對應功能摘要

優先順序：

1. 先修正會誤導協作者的過時描述
2. 再補充目前版本的功能與限制
3. 不追求一次把所有文件寫到完美

## commit / push 節奏

建議在以下條件都滿足後再提交：

1. 規格明確
2. 功能實作完成
3. 最小驗證完成
4. 驗收 thread 明確判定可提交
5. 文件與實作已同步

可搭配 [commit-workflow.md](commit-workflow.md) 一起使用。

## 目前實務建議

1. 一輪只做一個小功能或一個回修主題
2. 不把新功能與回修混在同一輪
3. 若規格有缺口，先回報缺口與影響，不自行擴張
4. 若文件尚未更新，不要把功能描述成已完整收斂
