# 第一階段功能規格：FastAPI 專案基礎骨架初始化

## 1. 功能名稱

第一階段 - FastAPI 專案基礎骨架初始化

## 2. 目標

建立本專案的最小可運行基礎架構，讓後續股票查詢、資料庫、模擬交易等功能有清楚的放置位置與啟動方式。

本階段重點不是實作商業功能，而是先建立：

1. 可啟動的 FastAPI 專案
2. 基本頁面與模板結構
3. 靜態資源結構
4. 基本設定檔與啟動說明

## 3. 功能範圍

本次要完成的內容：

1. 建立 `app/` 主程式結構
2. 建立 `app/main.py` 作為 FastAPI 入口
3. 建立基本首頁路由
4. 建立 `templates/` 與首頁模板
5. 建立 `static/` 與基本樣式檔
6. 建立 `config.py` 或基本設定檔結構
7. 建立 `.env.example`
8. 建立 `requirements.txt`
9. 建立 `README.md` 初稿，至少能說明安裝與啟動方式

## 4. 不包含項目

本次刻意不做的內容：

1. 股票資料 API 串接
2. 資料庫商業邏輯
3. 模擬交易功能
4. AI 預測功能
5. 使用者登入功能
6. 複雜前端畫面或動畫

## 5. 使用者操作流程

1. 開發者建立並安裝專案依賴
2. 開發者啟動 FastAPI 專案
3. 在瀏覽器開啟首頁
4. 頁面顯示專案名稱與啟動成功訊息

## 6. 輸入資料

本次功能主要輸入為：

1. 啟動指令
2. 環境設定值
3. 基本模板資料

不涉及外部股票資料輸入。

## 7. 輸出結果

本次完成後，系統應輸出：

1. 一個可啟動的 FastAPI 應用
2. 一個可開啟的首頁畫面
3. 一套清楚的專案骨架
4. 一份可供後續安裝與啟動的 README 初稿

## 8. 涉及頁面

預計涉及頁面：

1. 首頁

## 9. 涉及 API

本次預計至少包含：

1. 首頁路由
2. 基本健康或啟動成功頁面回應

## 10. 涉及資料表

本次不強制建立正式資料表。  
若未來要先預留 `data/` 目錄或資料庫初始化檔，可以只保留結構，不實作商業資料表。

## 11. 涉及檔案

建議新增或修改：

1. `app/main.py`
2. `app/config.py`
3. `app/routers/`
4. `app/templates/base.html`
5. `app/templates/index.html`
6. `app/static/css/style.css`
7. `.env.example`
8. `requirements.txt`
9. `README.md`

## 12. 驗收標準

以下條件達成才算完成：

1. 專案可正常啟動
2. FastAPI 應用可正常執行
3. 瀏覽器可開啟首頁
4. 首頁可顯示專案名稱或啟動成功訊息
5. 模板與靜態資源結構已建立
6. `README.md` 已提供基本安裝與啟動說明
7. 本輪修改能對應單一功能提交

## 13. 測試方式

建議最小必要驗證方式：

1. 手動啟動應用
2. 瀏覽器開啟首頁
3. 確認首頁內容正常顯示
4. 確認靜態樣式有被載入
5. 確認 README 中的啟動方式可理解

## 14. 風險與注意事項

1. 若一開始就引入過多套件，會增加環境混亂風險
2. 若前端設計花太多時間，會偏離本階段目標
3. 若目錄拆分過細，對第一階段學習反而會增加負擔
4. 目前應優先追求可啟動、可理解、可擴充

## 15. 學習重點

本次功能建議複習：

1. Python 專案結構
2. FastAPI 基本啟動
3. 路由概念
4. Jinja2 模板基礎
5. 靜態資源引用方式
6. 虛擬環境與套件管理
7. README 基本撰寫方式

## 16. Commit 切分建議

若要拆成多個 commit，可考慮：

1. `feat: initialize FastAPI project structure`
2. `feat: add homepage template and static assets`
3. `docs: add setup instructions and project bootstrap spec`

若本輪內容較小，也可整併成一個功能 commit。
