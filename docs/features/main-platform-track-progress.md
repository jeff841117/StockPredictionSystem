# `main` 平台主線進度

本文件用來追蹤目前 `main` 這條主線的整體平台進度。  
這裡的 `main` 指的是：

1. 作品集展示主線
2. 可對外展示的研究 / 模擬交易系統主線
3. 與 `daily-use` 可分開部署、可走不同網域的主版本

補充前提：

1. `main` 與 `daily-use` 可部署在不同網域
2. GitHub 仍維持同一個 repository
3. 以不同 branch 管理
4. 先完成 `main`，再回頭調整 `daily-use`
5. 若 `main` 有些內容不適合直接放進 `daily-use`，可只同步共用部分

---

## 1. 主線定位

`main` 目前定位為：

1. 作品集展示優先
2. 真實資料顯示優先
3. 研究工作台體驗優先
4. 工程完整度與可展示性優先

目前明確不要求：

1. 真實交易
2. 券商串接
3. 高頻個人化操作
4. 重度日常交易工作台

`main` 的資料方向：

1. 可以接真實數據 API 顯示
2. 不需要做真實交易流程

---

## 2. 設計參考方向

目前前端版面主要參考方向：

1. [TradingView Screener](https://tw.tradingview.com/screener/)
2. [TradingView TWSE:2330](https://tw.tradingview.com/symbols/TWSE-2330/)

使用方式：

1. 參考資訊密度
2. 參考 workspace 節奏
3. 參考研究頁閱讀順序
4. 參考 screener / symbol page 的產品感

不是要做的事情：

1. 不逐項照抄 TradingView 功能
2. 不假裝做成即時全市場選股平台
3. 不把頁面做得只有視覺像，但互動和資料能力不匹配

目前已知前端問題：

1. 仍有一些版面不夠貼近使用者習慣
2. 某些區塊仍偏死板
3. 某些呈現仍有 AI 感過重的問題

這些應持續列入 `main` 的優化方向。

## 4. 未完成功能

以下是 `main` 目前仍未完成、且建議納入平台主線的項目。

### 高優先

1. AI 研究報告最小版
說明：建立 `AI Forecast / Deep Analysis` 的最小資料流程與展示頁，但先定位為研究報告，不描述成投資建議或即時下單系統。

2. 多角色分析輸出層
說明：讓同一份報告可分成 `技術面 / 市場情境 / 風險 / 結論` 或 `交易員 / 新聞分析師 / 風控` 類角色輸出。

3. 研究資料來源擴充
說明：除了目前 TWSE 歷史價格外，開始盤點可展示且可合法使用的基本面、新聞、指數或市場摘要來源。

1. Swagger / OpenAPI 展示
說明：補齊 API 展示入口，提升後端可展示性。

2. API 分層整理
說明：讓目前頁面導向 + 交易導向 + 查詢導向的 API 結構更清楚。

3. Docker / deployment 文件
說明：你已有 Cloudflare 與自購網域，這一塊很適合補成可展示的部署能力。

4. CI 測試流程
說明：把目前測試轉成較正式的自動化流程展示。

### 中優先

5. AI 報告生成工作流
說明：建立「收集資料 -> 整理 prompt context -> 呼叫模型 -> 產出報告 -> 顯示結果」的最小工作流。

6. 模型輸出成本與快取策略
說明：若要真的對外展示 AI 研究報告，需要決定哪些結果即時生成、哪些結果可快取重用。

5. 更完整的錯誤處理與監控
說明：提升系統穩定性、故障辨識與展示可信度。

6. 操作日誌 / audit log
說明：目前已完成最小資料層記錄，涵蓋註冊、登入、登出、收藏新增 / 刪除、BUY / SELL；尚未提供查詢頁、篩選與匯出能力。

### 較後期

7. 即時或準即時市場資料層
說明：若之後真的要做更接近即時的市場判讀，需先補資料來源策略與資料更新節奏。

7. 權限管理
說明：目前已完成最小 `admin / user` 角色區分，並先套用到 audit log 管理頁與 admin API；尚未進入完整角色矩陣、權限編輯 UI 或細緻授權模型。

9. PostgreSQL + migration
說明：若平台要進入多使用者、audit log、部署穩定化，這一塊會變合理升級點。

## 4.1 參考平台可實作性判讀

根據你提供的參考方向，這套平台拆成幾層來看最合理：

1. 前端布局
可實作。首頁持倉資訊、報酬率、研究入口、AI 報告閱讀頁、左側導覽研究工作台，這些都屬目前專案可以承接的 UI / UX 升級。

2. AI 預測與深度技術分析
可實作，但第一版應定位為「研究報告生成」，不是高可信度預測引擎。

3. 多角色辯論分析
可實作，但建議第一版先做「單模型分角色輸出 + 系統總結」，不要一開始就做真正多 agent 長流程自治協作。

4. 即時股票、基本面、新聞、市場情境全量整合
可做，但目前不是立即可落地項目，因為專案現況仍以單一股票歷史價格與投資組合資料為主，缺少多來源資料流、排程、快取與資料品質控管。

結論：

- `main` 適合做成「可展示的 AI 股票研究平台」
- 不適合在現階段把對外敘事直接寫成「即時 AI 交易判斷平台」

## 5. 可以再優化功能

這些不是「沒做」，而是「已有第一版，但還能更成熟」。

1. 首頁與個股頁的前端閱讀節奏
目前狀態：已進入 workspace 風格。
可再優化方向：再降低死板感，讓區塊更貼近真實研究工具閱讀慣性。

2. UI 視覺語言的 AI 感
目前狀態：已有明顯改善。
可再優化方向：減少平均化卡片感、增加真正有主次的資訊排布。

3. API 錯誤訊息一致性
目前狀態：已有最小錯誤處理。
可再優化方向：統一頁面錯誤、API 錯誤、外部依賴錯誤分類。

4. 測試分層
目前狀態：已有 unit + smoke。
可再優化方向：再分清楚 unit / integration / external smoke。

5. 文件入口
目前狀態：README 與 docs 已有整理。
可再優化方向：讓部署、API、架構、驗收入口更直覺。

## 6. 分期建議

以下輪次是「從現在開始」往後排。

### 第 1 輪

1. AI 研究報告最小版
2. 研究資料來源擴充
3. Swagger / OpenAPI 展示
4. API 分層整理

原因：

1. 先把「AI 股票分析平台」真正做出最小閉環
2. 同時保留作品集展示力
3. 風險仍低於直接做登入 / 資料庫升級 / 即時系統

### 第 2 輪

1. 多角色分析輸出層
2. AI 報告生成工作流
3. Docker / deployment 文件
4. CI 測試流程

原因：

1. 這輪把報告從靜態展示推進到有生成流程的研究系統
2. 再補工程完整度與部署說服力

### 第 3 輪

1. 模型輸出成本與快取策略
2. 更完整的錯誤處理與監控
3. 操作日誌 / audit log

原因：

1. 這輪開始解決 AI 平台真正可展示、可穩定跑的工程問題
2. 但仍不必先進多使用者

### 第 4 輪

1. 即時或準即時市場資料層
2. 帳號登入
3. 權限管理

原因：

1. 這輪才開始碰更重的資料與使用者模型
2. 避免太早承諾即時與多使用者，導致主線失焦

### 第 5 輪

1. PostgreSQL + migration

原因：

1. 如果第 4 輪真的開登入與權限，這輪就很合理
2. 如果只是作品集展示短期優先，SQLite 可先保留

## 7. 關於 SQLite / PostgreSQL 的判斷

目前建議：

1. 若只看「是否影響面試」，SQLite 可以先繼續使用
2. 若之後要做登入、權限、audit log、較正式部署，PostgreSQL 會更合理

現階段判定：

1. PostgreSQL 不是現在必做
2. 但它已是 `main` 平台主線的中後期候選

## 8. 目前進度判讀

可以這樣看：

- MVP 第一版：已完成
- 前端第二階段第一波：已完成
- `main` 平台級補強：尚未正式開始

目前整體判定：

1. `main` 的產品雛形已完成
2. `main` 的作品集展示已經有基底
3. 接下來應從「平台完整度」補起，而不是立刻轉向真實交易

## 9. 更新規則

後續更新本文件時，建議遵守：

1. 新功能完成後，從「未完成功能」移到「已完成功能」
2. 只做第一版但還能精修的，移到「可以再優化功能」
